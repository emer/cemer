Since last checkin:

DND/CLIP

* dnd/clip standard ops fixed/tested
* added all the ops to the main and context menus
* group operations now working, both inproc and ext
* added new Assign op -- paste/drop an obj to assign
  it to another one -- you can assign from any 
  class >= the class being assigned
* note: Duplicate is in limbo, pending implementation
  (flags/menus are there, just needs impl) -- issue
  is that Dupe is a weird hybrid of a SRC and DST op
  and our paradigm doesn't well support it, but it
  is probably hackable (like Delete, but Delete is
  way easier, because there is no real operation to
  do, vs. Dupe, which requires making then inserting
  new objects)

Issues:

* something not right when using main menu to paste stuff
  into a list -- ticket 387

USER INFORMING

* all user messages now go through standard system
  message dialogs
* taMisc:: apis are unchanged (should normally use
  those)
* taiChoiceDialog is now a subclass of QMessageBox
* there is now a taMisc::Confirm -- this should be
  used for all popup messages that just require Ok
  * i replace all the Choice(blah, Ok) calls with
   this -- it is not a big deal, but is more standard
  since the icon is a simple message, not a question
* error dialogs now have a Copy To Clipboard
  
I have hoped to keep the Choice guys working the same,
but there could be some differences, particularly,
the system will now sometimes reorder the buttons
in the dialog, but note that the *chs* values will
always be correct (the order you supplied them)

WARNING: there is a chance the order of buttons on
 your favorite confirmation dialog could have changed,
 depending on your platform -- when you use this new
 release, pay small attention to choice dialogs
 when they first come up, before hitting the button
 in the place you are used to!! (sorry, there was
 nothing i could do about this, it is baked into
 Qt)

I eliminated most of the few remaining manual 
calls to taiChoiceDialog,
and replaced with calls to taMisc::Choice -- it is
not forbidden to use the dialog directly, but please
note that I've changed the api so that the msg and
the button list are separate -- but apart from that
it is similar to before.

You can use the dialogs directly, particularly if
you want to supply a wintitle -- note the order of
the args has changed slightly -- the title is now
after the msg, since the title is optional -- it 
defaults to the app name (per MS guidelines)

PROGRAMS

* very major change to how Method args are handled:
 the default value is NO LONGER set into the arg
 itself, it is only DISPLAYED for the user reference --
 the reason is that the combination of maketa
 parsing along with css just does not properly handle
 default arguments -- so instead, they are tracked
 but left blank -- if you want all the def args to
 use their defaults, do nothing -- the call will simply
 not pass those args, and the defaults will get used
 -- if you *want* to use some defaults, THEN a non-default
 after those defaults, then you will have to manually
 copy in the default values to the ones before your
 non-default ex:
 
 MyMeth(int a, int b=1, int c=2, int d=3)
 
 if you want the defaults, do nothing, and just set a;
 but say you wanted d to be 4, you'd have to do this:
 a=val, b=1, c=2, d=4
 
The handling of expr value itself has not changed at all,
and any values that are in saved programs will load as
before, so anything that is currently working, will work
identically -- you don't need to do anything -- this only 
applies to when you add a new MethodCall or StaticMethodCall

* method args are now checked to insure required values
  are supplied
  
* note that there is an issue with how our system parses
  default arguments from the source code, such that 
  there can be ambiguities, particularly with string 
  arguments, since we don't keep the quotation signs --
  so this will be an outstanding issue -- you just have
  to use some commonsense try to figure out what makes
  sense, ex. _nilString means the nilstring value, 
  not the literal string "_nilString"; for const char*
  args, 0 will typically mean NULL, not "0", etc.
  
* note that there is no way to resupply C++ default 
  arguments, such as default reference variables -- but
  these cases are fairly rare, and we will try to fix up
  the api to help avoid exposing these types of tricky
  methods to the user
  
note: i did test our standard projects, and they all
  seem to load and run fine
